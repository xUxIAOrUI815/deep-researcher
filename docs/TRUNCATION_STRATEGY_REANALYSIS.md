## 2. 当前截断策略的问题

### 2.1 当前策略

```python
truncated = markdown_text[:8000]  # 约5,300 tokens
```

### 2.2 覆盖率分析

| 截断阈值         | 覆盖率   | 说明          |
| ------------ | ----- | ----------- |
| 8,000字符 (当前) | \~64% | 36%的网页会丢失内容 |
| 15,000字符     | \~79% | 21%的网页会丢失内容 |
| 30,000字符     | \~90% | 10%的网页会丢失内容 |
| 50,000字符     | \~97% | 3%的网页会丢失内容  |

### 2.3 问题量化

**当前8000字符截断的问题**:

| 场景   | 占比  | 影响         |
| ---- | --- | ---------- |
| 完全覆盖 | 64% | 无问题        |
| 部分丢失 | 22% | 丢失10-50%内容 |
| 大量丢失 | 14% | 丢失50%以上内容  |

**对于深度调研场景**:

- 深度分析报告通常在10,000-50,000 tokens
- 这类文档占18%，是调研的主要目标
- 当前截断会导致关键数据丢失

## 3. 分层策略重新评估

### 3.1 原分层策略的问题

**原方案**:

```
≤ 10,000字符 → 不截断
≤ 30,000字符 → 智能截断
> 30,000字符 → 分块处理
```

**问题分析**:

| 问题         | 说明                               |
| ---------- | -------------------------------- |
| **阈值偏低**   | 10,000字符只覆盖64%，意味着64%的场景判断逻辑是多余的 |
| **分层过多**   | 对于大多数场景(79%)，只需要一个简单的阈值          |
| **复杂度高**   | 分块处理逻辑复杂，但只覆盖3%的场景               |
| **ROI不合理** | 为3%的极端场景投入大量开发成本                 |

### 3.2 基于数据的优化策略

**新方案: 简化为两层**

```
┌─────────────────────────────────────────────────────────────────────┐
│                    优化后的分层策略                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  第一层: 直接处理 (覆盖97%)                                          │
│  ├── 阈值: 50,000字符 (约33,000 tokens)                             │
│  ├── 策略: 智能截断至50,000字符                                     │
│  └── 覆盖: 97%的网页                                                │
│                                                                     │
│  第二层: 特殊处理 (覆盖3%)                                           │
│  ├── 阈值: > 50,000字符                                             │
│  ├── 策略: 分块处理或警告提示                                       │
│  └── 覆盖: 3%的极长文档                                             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 4. 分层策略的问题分析

### 4.1 分层判断的问题

| 问题       | 说明              | 影响     |
| -------- | --------------- | ------ |
| **判断时机** | 需要先获取完整内容才能判断长度 | 增加内存占用 |
| **判断开销** | 每次都需要计算长度       | 可忽略    |
| **边界效应** | 在阈值附近的文档可能被不同处理 | 轻微影响   |

### 4.2 分层处理的问题

| 问题       | 说明          | 解决方案    |
| -------- | ----------- | ------- |
| **逻辑复杂** | 需要维护多套处理逻辑  | 简化层数    |
| **测试成本** | 需要测试多种场景    | 聚焦主要场景  |
| **配置复杂** | 用户需要理解多个配置项 | 提供合理默认值 |

### 4.3 分块处理的问题

| 问题        | 说明            | 严重程度 |
| --------- | ------------- | ---- |
| **事实重复**  | 同一事实可能出现在多个块中 | 中    |
| **跨块事实**  | 一个事实可能被拆分到多个块 | 高    |
| **上下文丢失** | 分块后丢失全局上下文    | 高    |
| **成本增加**  | 多次API调用       | 中    |
| **时间增加**  | 串行处理增加延迟      | 中    |

**跨块事实问题示例**:

```
原文:
"台积电计划在2025年投资200亿美元建设2nm芯片产线，
预计2026年实现量产，届时将成为全球最先进的制程。"

分块1 (0-8000字符):
"台积电计划在2025年投资200亿美元建设2nm芯片产线，"

分块2 (8000-16000字符):
"预计2026年实现量产，届时将成为全球最先进的制程。"

结果: 一个完整事实被拆成两个不完整的片段
```

## 5. 推荐方案

### 5.1 方案对比

| 方案                | 覆盖率  | 复杂度   | 成本    | 推荐度   |
| ----------------- | ---- | ----- | ----- | ----- |
| A. 提高阈值至50,000字符  | 97%  | ⭐ 极低  | +200% | ⭐⭐⭐⭐⭐ |
| B. 两层处理(50K阈值+分块) | 100% | ⭐⭐ 低  | +250% | ⭐⭐⭐⭐  |
| C. 三层处理(原方案)      | 100% | ⭐⭐⭐ 中 | +300% | ⭐⭐⭐   |

### 5.2 最终推荐: 方案A + 可选分块

```python
class DistillerAgent:
    def __init__(
        self,
        ...
        max_content_length: int = 50000,  # 提高到50,000字符
        enable_chunking: bool = False      # 可选启用分块
    ):
        self.max_content_length = max_content_length
        self.enable_chunking = enable_chunking

    def _prepare_content(self, markdown_text: str) -> str:
        if len(markdown_text) <= self.max_content_length:
            return markdown_text
        
        if self.enable_chunking:
            return self._chunk_and_merge(markdown_text)
        
        return self._smart_truncate(markdown_text, self.max_content_length)
    
    def _smart_truncate(self, text: str, max_length: int) -> str:
        """在语义边界截断"""
        if len(text) <= max_length:
            return text
        
        truncated = text[:max_length]
        
        # 在段落边界截断
        last_para = truncated.rfind('\n\n')
        if last_para > max_length * 0.8:
            return truncated[:last_para]
        
        # 在句子边界截断
        for end_char in ['。', '.', '！', '!', '？', '?']:
            last_sent = truncated.rfind(end_char)
            if last_sent > max_length * 0.8:
                return truncated[:last_sent + 1]
        
        return truncated
```

## 6. 结论

### 6.1 分层是否合理？

**原分层策略不合理**，原因：

1. 阈值设置偏低，与真实数据分布不匹配
2. 层次过多，增加不必要的复杂度
3. ROI不合理，为少数场景投入过多

### 6.2 推荐策略

**简化为两层**:

- **主层**: 50,000字符阈值，覆盖97%场景
- **可选层**: 分块处理，覆盖剩余3%

### 6.3 实施优先级

1. **立即**: 提高阈值至50,000字符
2. **短期**: 实现智能截断
3. **可选**: 分块处理 (仅在确实需要时)
